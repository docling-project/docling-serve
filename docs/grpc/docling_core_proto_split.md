# docling-core vs docling-serve: Proto and API Definition Split

## Exploration summary (using `/work/docling-core`)

### docling-core today

| What | Where |
|------|--------|
| **API / schema source of truth** | Pydantic models in `docling_core/types/doc/` (e.g. `document.py`, `labels.py`, `base.py`) |
| **Generated API artifacts** | `docs/DoclingDocument.json`, `Generic.json`, `Record.json` (JSON Schema from Pydantic via `docling_core/utils/generate_jsonschema.py`) |
| **Proto** | None. No `.proto` files and no grpcio/protobuf in `pyproject.toml`. |

So in docling-core, “API definitions” = Pydantic types + generated JSON schema. There is no gRPC/protobuf surface there yet.

### docling-serve today

| What | Where |
|------|--------|
| **Core document proto** | `proto/ai/docling/core/v1/docling_document.proto` — mirrors DoclingDocument (and related types) for gRPC. |
| **Serve-specific proto** | `proto/ai/docling/serve/v1/docling_serve.proto`, `docling_serve_types.proto` — RPCs, `ConvertRequest`/`ConvertResponse`, `Source` (file/http/s3), etc. |
| **Generated code** | `docling_serve/grpc/gen/` — all pb2/pb2_grpc (core + serve) generated by `scripts/gen_grpc.py` from `proto/`. |
| **Usage** | `docling_serve.grpc` uses core pb2 for document on the wire; converter and schema validator map Pydantic ↔ proto. |

The serve proto imports the core document proto:  
`import "ai/docling/core/v1/docling_document.proto";`

---

## Recommendation: what lives where

### Move into docling-core (API definitions)

- **`proto/ai/docling/core/v1/docling_document.proto`**  
  This is the document-model API. It should live next to the canonical DoclingDocument definition (Pydantic in docling-core), so:
  - docling-core owns the document schema in both Pydantic and protobuf.
  - Other consumers (docling-serve, future gRPC clients, other languages) can depend on docling-core for the document contract.

Optional but useful:

- **Generated Python for the document proto**  
  e.g. `docling_core/proto/gen/ai/docling/core/v1/docling_document_pb2.py` (and no grpc service for that file). Then docling-core can expose something like:
  - `docling_core.proto.gen.ai.docling.core.v1.docling_document_pb2`
  and docling-serve would depend on that instead of generating the same from its own tree.

### Keep in docling-serve

- **`proto/ai/docling/serve/v1/*`** — RPC definitions, Convert request/response, Source types.
- **All gRPC server implementation** — `docling_serve/grpc/server.py`, mapping, schema validator, docling_document_converter, and any other serve-specific code.
- **Generation of serve-only stubs** — docling-serve continues to generate pb2/pb2_grpc for the *serve* protos only; for the document message it either:
  - uses the pb2 from docling-core (if docling-core ships generated code), or
  - runs protoc with `-I` pointing at docling-core’s proto dir and only generates serve protos (so the core proto is not duplicated in the serve repo).

### Dependency direction

- **docling-serve** depends on **docling-core** (already the case).  
- **docling-core** must not depend on docling-serve.  
- So: document proto and (optionally) its generated Python live in docling-core; serve proto and gRPC server stay in docling-serve.

---

## Implementation options

### Option A: Proto only in docling-core; docling-serve keeps generating core + serve

- Add to docling-core:
  - `proto/ai/docling/core/v1/docling_document.proto` (moved/copied from docling-serve).
- docling-serve:
  - Removes `proto/ai/docling/core/` from its repo.
  - In `scripts/gen_grpc.py`, add an include path to docling-core’s proto root (e.g. from `docling_core` package or a known path when developing with a local clone), and keep generating both core and serve into `docling_serve/grpc/gen/` so import paths stay the same.
- Pros: Single source of truth for the document proto in docling-core; minimal change to docling-core (no new deps).  
- Cons: docling-serve still generates and ships the core pb2; other languages would need to pull the proto from docling-core separately.

### Option B: Proto + generated Python in docling-core (preferred for “API definitions live in docling-core”)

- docling-core:
  - Adds `proto/ai/docling/core/v1/docling_document.proto`.
  - Adds a small build step (e.g. script + grpcio-tools in dev/build) to generate Python into e.g. `docling_core/proto/gen/ai/docling/core/v1/docling_document_pb2.py`.
  - Exposes that module (e.g. `docling_core.proto` or `docling_core.grpc.gen`). Optional: add `grpcio`/`protobuf` as optional or normal dependency.
- docling-serve:
  - Drops `proto/ai/docling/core/` and generation of core pb2.
  - Imports: `from docling_core.proto.gen.ai.docling.core.v1 import docling_document_pb2` (or whatever public path docling-core chooses).
  - `scripts/gen_grpc.py` only compiles serve protos; protoc is run with `-I` pointing at docling-core’s proto dir so `import "ai/docling/core/v1/docling_document.proto"` in serve protos still resolves.
- Pros: One place for the document API (Pydantic + JSON Schema + proto + generated pb2); docling-serve only generates serve-specific code; other Python clients can use docling-core for the document proto.  
- Cons: docling-core takes on a (small) proto build and possibly grpcio/protobuf.

---

## Suggested next steps

1. **Align with docling-core maintainers** — Confirm they are willing to host `proto/ai/docling/core/v1/docling_document.proto` and, if so, whether they want to ship generated Python (Option B) or only the proto (Option A).
2. **If Option B:** In docling-core, add `proto/` layout, a `gen_grpc.py`-style script that generates only the document pb2 (no grpc service), and expose the generated module; add `protobuf` (and optionally `grpcio`) to `pyproject.toml` as needed.
3. **In docling-serve:** Remove core proto from repo; point gen script at docling-core’s proto dir; switch imports to docling-core’s document pb2 once available; keep all serve proto and server code unchanged in docling-serve.

This keeps “document API definition” in docling-core and “serve API and implementation” in docling-serve, with a clear dependency direction and a migration path that doesn’t break the current gRPC PoC.
